Understanding Text Diff: How It Works and When to Use It

A diff (short for "difference") is a representation of the changes between two versions of a text. It highlights exactly what was added, removed, or left unchanged. This concept is fundamental to version control systems like Git, but it's equally useful in everyday tasks like comparing document revisions, validating configuration file changes, or reviewing editorial edits.

Three Diff Granularity Levels

The granularity at which you compare two texts dramatically affects the clarity of the result:

  • Line diff: Each line is treated as an atomic unit. If a single word changes in a 100-word paragraph, the entire line is marked as removed and the modified version is shown as added. This is the standard mode for code diffs (git diff, diff -u) because code is inherently line-oriented.
  • Word diff: Within changed segments, individual words are highlighted. This reveals exactly which word was replaced, added, or removed — ideal for comparing prose, legal text, or documentation where the structure is sentences and paragraphs rather than code lines.
  • Character diff: Every single character change is highlighted. This is the finest granularity — perfect for spotting a missing comma, an extra space, a transposed letter, or a Unicode character that looks identical to an ASCII character but isn't.

How the Myers Diff Algorithm Works

Under the hood, our tool uses the Myers diff algorithm (via the diff.js library), which is the same algorithm used by Git. The algorithm works by finding the shortest edit script — the minimum number of insertions and deletions needed to transform one text into the other. This is equivalent to finding the Longest Common Subsequence (LCS) of the two inputs and marking everything not in the LCS as either an insertion or deletion.

The Myers algorithm is particularly efficient because it exploits the observation that most practical diffs are short — the two files are mostly similar, and only a small fraction of content differs. The algorithm's worst-case time is O((N+M)D) where D is the edit distance (number of changes), making it very fast when files are mostly identical.

Practical Use Cases

  • Code review: Compare a script before and after modifications to verify only the intended changes were made.
  • Legal document review: Compare two versions of a contract to see which clauses were added, removed, or modified between drafts.
  • Content editing: Review the changes an editor made to your article draft. Word-level diff is ideal here.
  • Configuration management: Compare two versions of a server config file to understand what changed during a deployment.
  • Plagiarism detection: Check if two texts are suspiciously similar by looking at what little differs between them.
  • Translation review: Ensure that all source text segments have corresponding translated segments and nothing was dropped.
  • Data validation: Verify that a data export today matches (or correctly differs from) yesterday's export.

Reading the Diff Output

The result panel uses a standard color convention:

  • Green lines are content present in the new text but not the original — these are additions.
  • Red lines are content present in the original text but not the new — these are deletions.
  • Gray lines are identical in both texts — these are unchanged context lines.

Frequently Asked Questions

Is the diff library accurate?
Yes. The tool uses the battle-tested diff.js library (version 5), which implements the Myers diff algorithm — the same algorithm used by Git. It produces minimal, accurate diffs identical to what you'd see running git diff or diff -u on the command line.
Can I compare more than two files?
This tool compares exactly two texts. For multi-way merges or three-way diffs, you'd need a dedicated merge tool. However, you can run multiple pairwise comparisons (A vs B, B vs C) to trace changes across versions.
Why does word diff sometimes show entire sentences as different?
Word diff splits text on whitespace boundaries. If a sentence is fully rewritten with no common words, the entire sentence appears as one removal and one addition. This is technically correct — there are no common word subsequences to anchor the diff to.
Does this tool support exporting the diff as a patch file?
Currently, the "Copy Result" button copies the plain-text diff output. For a proper unified diff (.patch) format, you can manually copy the result — lines starting with "-" were removed and "+" were added. Full patch format export is a planned feature.
